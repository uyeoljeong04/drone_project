## 0-1. 그래서 드론이 왜 안 움직이나 — 원인은 정확히 2줄

**① world 파일** — `worlds/spartina_world.world:32`

```xml
<model name="drone_camera">
  <static>true</static>          ← 물리적으로 고정
  <pose>0 0 8 0 1.5708 0</pose>
```

**② 런치 파일** — `launch/spartina_nav2.launch.py:82`

```python
static_tf_camera = Node(
    executable='static_transform_publisher',   ← 좌표도 영구 고정
    arguments=['--x','0.0','--y','0.0','--z','8.0', ...])
```

`static_transform_publisher` 는 **한 번 발행하고 영원히 안 바뀌는** TF다.
Gazebo 에서 드론을 마우스로 끌어도 이 TF 는 "(0,0,8)"이라고 계속 우기고,
그래서 투영 좌표가 틀어진다.

> **핵심**: 투영 로직은 손댈 게 없다. TF 만 살아 움직이게 만들면
> 그 아래 파이프라인(locator → goal_relay → Nav2)은 **코드 한 줄 안 바꾸고** 그대로 동작한다.
> 처음부터 TF2 기반으로 짜놨기 때문에 얻는 이득이다.

# Phase 1 — 정적 TF를 동적 TF로 교체

**이번 단계 목표: 드론을 손으로 아무 데나 옮겨도 좌표가 정확히 나오게 만든다.**

## 새로 만든 파일 (기존 파일은 건드리지 않음)

| 파일 | 역할 |
|---|---|
| `worlds/spartina_world_moving.world` | 드론 모델 kinematic 화 + `gazebo_ros_state` 플러그인 추가 |
| `scripts/drone_tf_broadcaster.py` | Gazebo 실제 드론 pose → `map→drone_camera_optical` TF 실시간 발행 |
| `launch/spartina_moving.launch.py` | 위 두 개를 쓰는 런치. static camera TF 제거 |

기존 `spartina_world.world`, `spartina_nav2.launch.py` 는 그대로 살아있으니
안 되면 언제든 예전 방식으로 복귀 가능하고, 팀원 작업도 안 깨진다.

## world 파일 변경 내용 (3가지)

1. **`<static>true</static>` → `<static>false</static>` + 링크에 `<kinematic>true</kinematic>`**
   static 모델은 `SetEntityState` 로 옮겨도 pose 가 갱신되지 않는 경우가 있다.
   kinematic 은 중력·충돌력을 안 받으면서 pose 설정은 확실히 먹는다.

2. **`<gravity>false</gravity>` + `<inertial>` 추가**
   static 을 풀면 중력을 받아 떨어지므로 중력을 끈다.
   non-static 링크는 inertial 이 없으면 Gazebo 가 경고를 낸다.

3. **`libgazebo_ros_state.so` 플러그인 추가** ← 이게 없으면 아무것도 안 됨

   | 제공하는 것 | 쓰는 곳 |
   |---|---|
   | 토픽 `/gazebo/model_states` | Phase 1 — 드론 실제 pose 읽기 |
   | 서비스 `/gazebo/set_entity_state` | Phase 2 — 드론 이동시키기 |

## 좌표계 — 여기가 유일하게 헷갈리는 부분

두 프레임의 축 방향이 다르다.

| 프레임 | 규약 |
|---|---|
| **Gazebo 카메라** | x = 앞(보는 방향), y = 왼쪽, z = 위 |
| **ROS 광학(optical)** | x = 오른쪽, y = **아래**, z = **앞** ← `pixel_to_ground` 가 쓰는 규약 |

이 둘 사이는 **항상 똑같은 고정 회전**이고, 쿼터니언으로 `(-0.5, 0.5, -0.5, 0.5)` 다.
`drone_tf_broadcaster.py` 는 이렇게 계산한다:

```
발행할 TF 자세 = (Gazebo 모델의 현재 자세) ⊗ Q_CAM2OPT
```

### 이 값이 맞는지 이미 수치로 검증했다

world 의 모델 pose `rpy = (0, 1.5708, 0)` 에 `Q_CAM2OPT` 를 합성하면

```
R = [[ 0, -1,  0],
     [-1,  0,  0],
     [ 0,  0, -1]]
```

이 나오는데, 이건 **기존 런치의 static TF 쿼터니언 및
`obstacle_locator_node.py:106` 의 `R_HARDCODED` 와 정확히 일치**한다.

추가로 드론을 (3, −2, 8) 에 두고 화면 중앙 픽셀을 투영하면 정확히 (3, −2, 0) 이 나온다.

→ **드론이 제자리에 있을 때는 기존과 100% 동일하게 동작하고, 움직이면 그때부터 따라간다.**
   즉 이 교체로 기존 Week4 결과가 깨질 일은 없다.

# Phase 2 — 드론 자율 비행 (지그재그 탐사)

**이번 단계 목표: 드론이 스스로 탐사 영역을 훑는다.**

## 파일

`scripts/drone_survey_node.py` — lawnmower(지그재그) 경로를 자동 생성하고
`/gazebo/set_entity_state` 로 드론을 부드럽게 이동시킨다.

## 비행 물리를 쓰지 않는 이유 (보고서에 쓸 논리)

쿼드로터 동역학(PX4 SITL 등)은 그 자체로 별도 프로젝트다.
우리 검증 대상은 **탐지 → 좌표 변환 → 로봇 주행** 파이프라인이지 비행 제어가 아니다.
따라서 "드론 위치는 이상적으로 주어진다"고 가정하고 위치를 직접 보간해 옮긴다.

→ 보고서에는 **"비행 제어는 연구 범위 밖, 위치는 이상적으로 주어진다고 가정"** 으로 명시.
   이렇게 해두면 한계로 지적당하는 게 아니라 **의도된 범위 설정**이 된다.

## 레인 간격 정하는 법 (근거 있게)

고도 8m, HFOV 60° 기준 지상 가로 시야폭:

```
2 × 8 × tan(30°) ≈ 9.24 m
```

30% 겹침을 주려면 `lane_spacing ≈ 6.5 m`.
다만 지금 월드는 좁으니 (±4m 정도) `lane_spacing:=3.0` 으로 시작.

## 실행

```bash
# T3  탐사 비행 시작
python3 ~/drone_project/scripts/drone_survey_node.py --ros-args \
  -p use_sim_time:=true \
  -p area_x_min:=-4.0 -p area_x_max:=4.0 \
  -p area_y_min:=-3.0 -p area_y_max:=3.0 \
  -p lane_spacing:=3.0 -p altitude:=8.0 -p speed:=0.8 -p hold_sec:=3.0
```

## 설계 포인트: 웨이포인트에서 멈춘다

각 웨이포인트에 도착하면 `hold_sec` 만큼 정지하고, 그 상태를
`/drone/survey_state` 에 `HOLD` 로 발행한다.

**Phase 4 탐지기를 "HOLD 일 때만 탐지"하게 만들면 모션블러와 동기화 오차를 동시에 피한다.**
초반에는 이 방식으로 확실하게 만들고, 연속 탐지는 나중에 도전.

- 이동 중 아무 지점에서나 기둥을 클릭 → 여전히 (2.0, 0.0)

# Phase 3 — timestamp 동기화 ⚠️ 반드시 터질 버그

**Phase 2 를 하고 나면 오차가 갑자기 커진다. 원인은 이거다. 미리 알고 가자.**

## 문제

`/detection/pixel` 이 `geometry_msgs/Point` 다. **timestamp 필드가 없다.**

`obstacle_locator_node.py:186` 은 이렇게 조회한다:

```python
tf = self.tf_buffer.lookup_transform(self.world_frame, frame_id, rclpy.time.Time())
#                                                              ↑ "지금 이 순간의 최신 TF"
```

드론이 고정일 땐 아무 문제 없다. 하지만 움직이면:

```
시각 t     : 영상 촬영 (이때 드론은 A 위치)
시각 t+δ   : 탐지 결과 도착 → 그런데 TF 는 "지금"(=B 위치) 것을 가져옴
             → A 에서 본 픽셀을 B 의 pose 로 투영 = 틀림
```

**고도 8m, 속도 1 m/s, 지연 0.1초 → 약 10cm 오차.**
현재 계산 오차가 1cm 인 걸 생각하면 **이게 단숨에 최대 오차원이 된다.**

## 해결

1. `/detection/pixel` 을 `geometry_msgs/PointStamped` 로 변경
2. 발행하는 쪽(`click_pixel_node.py` / 탐지기)이 **영상 헤더의 timestamp 를 그대로** 실어줌
3. `obstacle_locator_node.py` 를 이렇게 수정:

```python
def on_detection_pixel(self, msg):        # PointStamped
    self.current_pixel = (msg.point.x, msg.point.y)
    self.stamp = msg.header.stamp          # ← 영상이 찍힌 시각을 기억

# lookup 할 때
tf = self.tf_buffer.lookup_transform(
    self.world_frame, frame_id,
    rclpy.time.Time.from_msg(self.stamp),  # ← 그 시각의 TF 를 조회
    timeout=rclpy.duration.Duration(seconds=0.2))
```

`tf_buffer` 는 과거 TF 를 버퍼에 보관하므로 과거 시각 조회가 가능하다.

## ✅ Phase 3 합격 기준

드론이 **이동 중일 때** 기둥을 클릭해도 오차가 Phase 1 수준(수 cm)을 유지.
수정 전/후 오차를 표로 비교하면 그대로 보고서 자료가 된다.

> 이 비교 실험은 발표에서 강력하다. "동기화를 안 하면 10cm, 하면 1cm" —
> 문제를 예측하고 해결했다는 증거이기 때문.

# Phase 4 — 탐지기 연동 + 다중 목표

> ⚠️ 여기서부터는 아직 실제로 돌려보지 않은 계획이다.
> Phase 1~3 을 끝낸 뒤 상황을 보고 조정해야 한다.

## 4-1. 탐지기 학습

`갯끈풀 데이터셋/` 에 실제 드론 촬영 이미지(DJI)가 이미 있다.

- YOLOv8 로 학습. 클래스 1개(`spartina`)면 충분
- **주의**: 데이터셋은 실제 갯벌 사진이고 시뮬은 빨간 원기둥이다. 도메인이 완전히 다르다
- 현실적 선택지 두 가지:

  | 방식 | 내용 | 장단점 |
  |---|---|---|
  | **A. 시뮬용 색 검출** | 시뮬 안에서는 HSV 색 검출로 기둥 탐지 | 즉시 됨. 파이프라인 검증에 충분 |
  | **B. 실사진 YOLO** | 실제 데이터셋으로 YOLO 학습, 별도 평가 | 실사용 근거. 시뮬 연동은 안 됨 |

  **A 와 B 를 둘 다 하고 분리해서 보고하는 걸 추천.**
  "파이프라인은 A 로 검증, 탐지 성능은 B 로 검증" — 정직하고 방어하기 쉽다.

## 4-2. 중복 제거 (새로 생기는 문제)

드론이 움직이면 **같은 갯끈풀이 수십 프레임에 걸쳐 검출된다.**
전부 목표로 넣으면 로봇이 같은 자리를 반복 방문한다.

해결: map 좌표로 변환한 뒤 클러스터링해서 병합.

```
새 탐지 좌표 P 가 들어옴
  → 기존 타겟 리스트에서 거리 < merge_radius(예: 0.5m) 인 게 있나?
     있으면  → 그 타겟의 좌표를 이동평균으로 갱신, 관측 횟수 +1
     없으면  → 새 타겟으로 추가
  → 관측 횟수가 N회(예: 3) 이상인 타겟만 "확정"으로 승격
```

관측 횟수 조건을 두면 **1회성 오탐이 자동으로 걸러진다.** 공짜로 얻는 정확도다.

## 4-3. 다중 목표 순회

확정된 타겟들을 로봇이 순서대로 방문.

- 순서: 가장 가까운 것부터(greedy) 로 시작. 최적 경로(TSP)는 과욕
- 각 타겟 도착 판정 후 다음 타겟으로 Nav2 goal 재발행
- `goal_relay_node.py` 를 "타겟 리스트 관리 + 순차 발행" 으로 확장

## 4-4. 최종 통합 시나리오 (데모 목표)

```
1. 드론이 지그재그로 탐사 영역 비행
2. 갯끈풀 3개를 각각 다른 위치에서 탐지
3. 중복 제거 후 확정 타겟 3개 확보
4. 로봇이 가까운 순서대로 3곳을 자율주행으로 방문
```
